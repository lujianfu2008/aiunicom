# 知识图谱功能使用说明

## 功能概述

知识图谱功能通过 LLM（大语言模型）自动从文档中识别实体和抽取关系，构建知识图谱，并提供可视化展示和交互式探索功能。

---

## 核心功能

### 1. 实体识别

自动识别文档中的实体，包括：

| 实体类型 | 说明 | 示例 |
|---------|------|------|
| 问题 | 描述的问题现象 | 网络连接失败、系统崩溃 |
| 解决方案 | 解决问题的方法或步骤 | 重启服务、修改配置文件 |
| 系统 | 涉及的系统或平台 | CBSS系统、CRM系统 |
| 组件 | 涉及的组件或模块 | 数据库模块、认证服务 |
| 错误代码 | 错误代码或错误信息 | ORA-12154、500错误 |
| 配置项 | 配置参数或设置 | max_connections、timeout |
| 命令 | 操作命令或指令 | systemctl restart、netstat -an |
| 文件 | 涉及的文件或路径 | /etc/hosts、config.ini |
| 人员 | 涉及的人员或角色 | 系统管理员、DBA |
| 时间 | 时间点或时间段 | 2024-01-01、每天凌晨2点 |

### 2. 关系抽取

自动识别实体之间的关系，包括：

| 关系类型 | 说明 | 示例 |
|---------|------|------|
| 导致 | 问题导致的结果 | 网络故障 → 导致 → 服务不可用 |
| 解决 | 解决方案解决的问题 | 重启服务 → 解决 → 系统卡顿 |
| 属于 | 实体所属的分类 | MySQL → 属于 → 数据库 |
| 包含 | 实体包含的子项 | 系统问题 → 包含 → 网络问题 |
| 依赖 | 实体之间的依赖关系 | 应用服务 → 依赖 → 数据库 |
| 配置 | 配置项与系统的关系 | max_connections → 配置 → MySQL |
| 执行 | 命令与操作的关系 | systemctl restart → 执行 → 服务重启 |
| 生成 | 错误代码的生成来源 | 网络超时 → 生成 → ORA-12154 |
| 涉及 | 问题涉及的系统或组件 | 登录失败 → 涉及 → 认证服务 |
| 关联 | 实体之间的关联关系 | 问题A → 关联 → 问题B |

### 3. 知识图谱可视化

使用 vis.js 提供交互式图谱展示：

- **节点**：表示实体，不同类型用不同颜色区分
- **边**：表示关系，箭头指向关系方向
- **交互**：支持缩放、拖拽、点击查看详情

## 使用方法

### 1. 访问知识图谱

1. 启动系统后，点击顶部标签页的"知识图谱"
2. 系统会自动加载知识图谱统计信息

### 2. 加载图谱

点击"加载图谱"按钮，系统会显示所有实体和关系的可视化图谱。

### 3. 从文本抽取知识

1. 点击"从文本抽取"按钮
2. 在弹出的对话框中输入要分析的文本
3. 确认后，系统会调用 LLM 进行实体识别和关系抽取
4. 抽取完成后，自动刷新图谱

### 4. 批量抽取知识

1. 点击"批量抽取"按钮
2. 输入要处理的文档数量（建议不超过10个）
3. 确认后，系统会从知识库中选择文档进行批量抽取
4. 抽取过程可能需要较长时间，请耐心等待

### 5. 搜索实体

1. 在搜索框中输入关键词
2. 点击"搜索"按钮
3. 系统会显示匹配的实体列表

### 6. 查看实体详情

1. 在左侧实体列表中点击实体
2. 或在图谱中点击节点
3. 系统会显示该实体的详细信息和相关关系

### 7. 清空图谱

点击"清空图谱"按钮可以删除所有实体和关系数据（需二次确认）。

## API 接口

### 1. 获取统计信息

```http
GET /api/kg/stats
```

**响应示例**：
```json
{
  "entity_count": 150,
  "relation_count": 230
}
```

### 2. 获取实体列表

```http
GET /api/kg/entities?type=问题&keyword=网络&limit=20
```

**参数**：
- `type`：实体类型（可选）
- `keyword`：搜索关键词（可选）
- `limit`：返回数量限制（默认50）

**响应示例**：
```json
{
  "total": 10,
  "entities": [
    {
      "id": "abc123",
      "name": "网络连接失败",
      "type": "问题",
      "properties": {},
      "created_at": "2024-01-01T12:00:00"
    }
  ]
}
```

### 3. 获取实体详情

```http
GET /api/kg/entity/{entity_id}
```

**响应示例**：
```json
{
  "entity": {
    "id": "abc123",
    "name": "网络连接失败",
    "type": "问题"
  },
  "relations": [
    {
      "id": "rel001",
      "source_id": "abc123",
      "target_id": "def456",
      "type": "导致"
    }
  ]
}
```

### 4. 获取图谱数据

```http
GET /api/kg/graph?entity_id=abc123&depth=2
```

**参数**：
- `entity_id`：中心实体ID（可选，不传则返回全部）
- `depth`：扩展深度（默认2）

**响应示例**：
```json
{
  "nodes": [
    {
      "id": "abc123",
      "label": "网络连接失败",
      "group": "问题"
    }
  ],
  "edges": [
    {
      "id": "rel001",
      "from": "abc123",
      "to": "def456",
      "label": "导致"
    }
  ]
}
```

### 5. 从文本抽取知识

```http
POST /api/kg/extract
Content-Type: application/json

{
  "text": "网络连接失败导致系统无法访问，解决方案是检查网络配置。",
  "doc_id": "doc001"
}
```

**响应示例**：
```json
{
  "success": true,
  "entities_count": 3,
  "relations_count": 2,
  "saved_entities": 3,
  "saved_relations": 2
}
```

### 6. 批量抽取知识

```http
POST /api/kg/batch_extract
Content-Type: application/json

{
  "limit": 10
}
```

**响应示例**：
```json
{
  "success": true,
  "total_documents": 10,
  "processed_documents": 10,
  "total_entities": 45,
  "total_relations": 38,
  "saved_entities": 45,
  "saved_relations": 38,
  "failed_documents": []
}
```

### 7. 清空知识图谱

```http
POST /api/kg/clear
```

**响应示例**：
```json
{
  "success": true
}
```

## 技术架构

### 1. 数据存储

使用 Redis 存储知识图谱数据：

- **实体存储**：`kg:entity:{entity_id}`
- **关系存储**：`kg:relation:{relation_id}`
- **类型索引**：`kg:entity_index:type:{type}`
- **名称索引**：`kg:entity_index:name:{name}`
- **关系索引**：`kg:relation_index:source:{entity_id}`

### 2. 核心模块

| 模块 | 文件 | 功能 |
|------|------|------|
| 数据模型 | `knowledge_graph.py` | 实体、关系的数据模型和存储 |
| 知识抽取 | `knowledge_extractor.py` | 使用 LLM 进行实体识别和关系抽取 |
| API 接口 | `api_server.py` | 提供 RESTful API 接口 |
| 可视化 | `api_server.py` | 使用 vis.js 进行图谱可视化 |

### 3. 依赖库

- **Redis**：数据存储
- **vis.js**：图谱可视化
- **LLM**：知识抽取（智谱 AI 或本地模型）

## 使用建议

### 1. 抽取策略

- **小批量测试**：先用少量文档测试抽取效果
- **逐步扩大**：确认效果良好后再批量抽取
- **定期更新**：新增文档后及时抽取知识

### 2. 质量控制

- **人工审核**：重要实体和关系需要人工确认
- **去重处理**：系统会自动去重相同实体
- **关系验证**：检查关系的合理性

### 3. 性能优化

- **限制数量**：批量抽取时限制文档数量
- **分批处理**：大量文档分批处理
- **缓存结果**：已抽取的结果会缓存

## 注意事项

### 1. LLM 调用

- 知识抽取需要调用 LLM，会产生 API 费用
- 批量抽取会多次调用 LLM，注意控制成本
- 建议在业务低峰期进行批量抽取

### 2. 数据质量

- LLM 抽取的知识可能不完全准确
- 建议对重要知识进行人工审核
- 可以通过清空图谱重新抽取

### 3. 性能影响

- 大量实体会影响图谱渲染性能
- 建议控制单个图谱的节点数量
- 可以通过搜索功能筛选实体

## 常见问题

### Q1: 为什么抽取不到实体？

**可能原因**：
- 文本内容过短或质量较差
- LLM 未正确识别实体
- 实体类型不在预定义范围内

**解决方法**：
- 检查文本内容质量
- 尝试不同的文本
- 联系管理员扩展实体类型

### Q2: 图谱显示不完整？

**可能原因**：
- 实体数量过多，部分被隐藏
- 关系复杂，布局不清晰

**解决方法**：
- 使用搜索功能筛选实体
- 点击实体查看局部图谱
- 调整图谱缩放级别

### Q3: 如何删除错误的实体？

**解决方法**：
- 目前需要清空整个图谱
- 后续版本会支持单个实体删除
