# 内容查找功能使用说明

## 功能概述

"内容查找"是一个**精确的内容搜索功能**，它会在所有文档的**内容字段**中搜索包含指定关键词的文档。

## 使用方式

### 1. Web 界面使用

#### 入口位置
- 打开 Web 界面：http://localhost:5000
- 点击"**精确查找**"标签页

#### 操作步骤
1. 在搜索框中输入关键词
2. 可以输入多个关键词，用**空格**分隔
3. 按回车键或点击"查找"按钮

#### 示例
```
输入：套餐变更 失败
结果：查找所有同时包含"套餐变更"和"失败"的文档
```

### 2. API 接口调用

#### 接口地址
```
GET /api/find_content
```

#### 请求参数
- `keywords` (必需): 关键词字符串，支持多种格式
  - 逗号分隔：`关键词 1，关键词 2`
  - 空格分隔：`关键词 1 关键词 2`

#### 请求示例
```bash
# 单个关键词
GET http://localhost:5000/api/find_content?keywords=套餐变更

# 多个关键词（逗号分隔）
GET http://localhost:5000/api/find_content?keywords=套餐变更，失败，报错

# 多个关键词（空格分隔）
GET http://localhost:5000/api/find_content?keywords=套餐变更 失败 报错
```

#### 响应示例
```json
[
  {
    "file_name": "2086733-客服问题 18643065775.txt",

    "problem_id": "2086733",
    "problem_type": "套餐变更",
    "content": "问题内容...",

    "created_time": 1234567890

```

## 搜索范围

内容查找会在以下字段中搜索关键词：

1. **content** - 文档主要内容
2. **solution** - 问题解决方案
3. **file_name** - 文件名

**注意**：搜索是**精确匹配**，关键词必须完整出现在上述字段中。

## 匹配规则

### 1. 所有关键词都必须匹配
```
输入：套餐 变更
结果：只返回同时包含"套餐"和"变更"的文档
```

### 2. 不区分大小写
```
输入：ERROR
结果：可以匹配到"error"、"Error"、"ERROR"
```

### 3. 支持中文分词
```
输入：套餐变更失败
结果：会自动分词为 ["套餐", "变更", "失败"]
```

## 与"语义搜索"的区别

| 特性 | 内容查找 | 语义搜索 |
|------|---------|---------|
| 匹配方式 | 精确关键词匹配 | 向量相似度匹配 |
| 搜索范围 | content + solution + file_name | 向量数据库 |
| 关键词要求 | 必须完全匹配 | 语义相关即可 |
| 适用场景 | 已知具体关键词 | 模糊查询、同义词查询 |
| 速度 | 较慢（需要扫描所有文档） | 较快（使用索引） |

### 使用建议

**使用"内容查找"的场景：**
- 知道具体的订单号、手机号
- 知道具体的错误代码
- 知道特定的业务术语
- 需要精确匹配某些关键词

**使用"语义搜索"的场景：**
- 不确定具体关键词
- 想要查找语义相关的内容
- 想要查找同义词、近义词
- 想要快速浏览相关内容

## 常见问题

### Q1: 为什么搜索结果为空？
- 检查关键词是否拼写正确
- 尝试使用更简短的关键词
- 尝试使用同义词
- 确认关键词确实存在于文档内容中

### Q2: 为什么搜索结果太多？
- 增加更多关键词进行筛选
- 使用更具体的关键词
- 使用更长的关键词短语

### Q3: 内容查找和语义搜索哪个更好？
- 没有绝对的好坏，取决于使用场景
- 已知具体信息 → 内容查找
- 模糊查询 → 语义搜索
- 建议两者结合使用

## 技术实现

### 架构设计原则

**核心原则：能放到 Redis 处理的，决不能放到 Python 处理。**

数据过滤、匹配等操作尽可能在 Redis 服务端完成，避免将大量数据传输到 Python 端再过滤。这样做的优势：
- 消除网络传输瓶颈（MB级 → KB级）
- 利用 Redis 内存计算的高效性
- 减少 Python 端的 CPU 和内存开销

### 后端实现

#### 调用链路
```
API请求 → /api/find_content → kb.find_by_content() → vector_store.find_by_content()
```

#### 三级匹配策略（优先级从高到低）

| 级别 | 方式 | 实现 | 说明 |
|------|------|------|------|
| 1 | Redis Lua 脚本 | `_find_by_content_lua()` | **首选**：在 Redis 服务端完成 SCAN + JSON.GET + 关键词匹配，只返回匹配的 key，再批量获取匹配文档数据 |
| 2 | Python 扫描 | `_find_by_content_python_scan()` | **fallback**：Lua 失败时使用，逐批 SCAN + pipeline JSON.GET 获取必要字段，Python 端过滤 |
| 3 | FT.SEARCH（已弃用） | - | 之前使用 RediSearch 全文索引 `*keyword*` 通配符查询，因中文分词不准且前导通配符导致超时而弃用 |

#### Lua 脚本实现（`_LUA_CONTENT_SEARCH`）

脚本在 Redis 服务端执行以下操作：
1. `SCAN` 遍历所有 `workorder:*` key
2. 对每个 key 获取 `$.content`、`$.solution`、`$.file_name` 三个字段
3. 在服务端做子串匹配（`string.find`，plain 模式，大小写不敏感）
4. 所有关键词必须全部匹配（AND 逻辑）
5. 只返回匹配的 key 列表

Python 端收到匹配 key 后，用 pipeline 批量获取完整文档数据返回给前端。

#### 性能对比

| 指标 | 优化前（Python扫描） | 优化后（Lua脚本） |
|------|---------------------|------------------|
| 匹配位置 | Python 端 | Redis 服务端 |
| 数据传输量 | 全部文档内容（~MB级） | 仅匹配文档（~KB级） |
| 网络往返 | 多次 SCAN + pipeline | 1次 Lua + 1次 pipeline |
| 预计耗时 | 1-3 秒 | 100-300ms |
